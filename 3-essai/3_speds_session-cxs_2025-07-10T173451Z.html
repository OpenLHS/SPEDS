3. Contexte général des tests

Le présent document a pour objectif de spécifier les essais qui détermineront si l’implémentation de la couche Session respecte la conception définie pour cette même couche.

La structure des essais est organisée de façon à ce que l’initiatrice de la session soit testée simultanément avec la partenaire de session en relayant les messages de la première vers la deuxième et inversement.

Conventions

La convention suivante est employée pour désigner des éléments testés partiellement, en raison des limites imposées par les entrées/sorties, lesquelles ne correspondent pas parfaitement aux frontières des éléments identifiés par les processus:

  • [PR0_X] : est couvert du début à la fin

  • [PR0_X~] : est couvert du début jusqu’à ce qu’on fasse une sortie

  • [~PRO_X] : est couvert à partir de l’entrée en milieu d’élément jusqu’à la fin de l’élément.

Bien que ce soit implicite, il est à noter que la méthode de passage des variables de l’environnement entre l’essai et le cas de test est au choix. Bien que certains éléments de l’environnement soient précisés en hypothèse , la liste des variables n’est pas exhaustive. De ce fait, le testeur peut ajouter les éléments nécessaires aussi.

Aussi, le nom utilisé pour les variables n’est pas aussi important que la cohérence entre la déclaration et l’utilisation de ces variables.

Enfin, quand on parle de mécanisme de trigger, on fait simplément référence au fait qu’il faut déclencher le test d’une façon ou d’une autre et comme certains appels demandent un déclenchement différé, on précise que le test devrait avoir un moyen de provoquer le début du test au moment propice.

6. Données globales / fonctions

Initialisation d’une entité

Nécessite:

  • Une paire de clef aléatoire de format RSA et de taille 4096

  • Un certificat aléatoire du format (X.509)

  • Les valeurs pour SPEDS fournit par l’environnement

Le détail des formats est une contrainte provenant de la conception.

Voir le readme de l’implémentation pour les détails d’initialisation.

Retour:

Retourne un hôte bien formé pouvant faire les opérations nécéssaires.

Informations de la session

Les informations pour une session sont composées de :

  • Un numéro de session unique généré par le processus (obligatoire, immutable) ;

  • L’identifiant de l’entité agissant comme initiatrice de la session (obligatoire, immutable) ;

  • La clé publique de l’entité agissant comme initiatrice de la session (obligatoire, immutable) ;

  • La clé privée de l’entité agissant comme initiatrice de la session (disponible seulement pour l’entité initiatrice de la session, immutable) ;

  • Le IRI de l’entité agissant comme initiatrice de la session (obligatoire, immutable) ;

  • L’identifiant de l’entité agissant comme partenaire de la session (obligatoire, immutable) ;

  • Le IRI de l’entité agissant comme partenaire de la session (obligatoire, immutable) ;

  • La clé d’authentification négociée SKAK (optionnelle, immutable) ;

  • L’identifiant du PGA (optionnelle, immutable) ;

  • La clé de chiffrement SDEK (optionnelle, immutable) ;

  • La charge utile à transmettre (pour le processus 1)

  • Le numéro de suivi du message en cours de traitement (optionnel, mutable) ;

  • Le nombre de messages traités avec succès durant la session (initialisé à 0, mutable).

Mécanismes de test prédéfinis

La configuration des mécanismes est de nature indicative ; il s’agit d’abord et avant tout d’être en mesure de déclencher les tests et de récupérer les entrées / sorties nécessaires pour le cas de test.

Les mécanismes déclencheurs :

Pour un PIDU, il s’agit de configurer un appel sur l’entité testée elle-même.
Pour un SIDU, il s’agit de configurer un message entrant depuis la couche inférieure.

Les mécanismes de collecte de données:

Pour un PIDU, il s’agit simplement de récupérer la sortie de l’appel. Pour un SIDU, il s’agit de capturer l’entrée sortante vers la couche inférieure.

10.1. Glossaire

[PDU]

Protocole Data Unit - Unité de données du protocole. Correspond à la valeur de la propriété message d’une IDU. C’est le message destiné à l’homologue de communication de la couche.

[PPDU]

Presentation Protocol Data Unit - PDU de la couche présentation.

[SPDU]

Session Protocol Data Unit - PDU de la couche session.

[IDU]

Interface Data Unit - Unité de données d’interface. Permet de passer des informations entre les couches

[PIDU]

Presentation Information Data Unit - IDU de la couche présentation.

[SIDU]

Session Information Data Unit - IDU de la couche session.

[SDU]

Service Data Unit - Unité de données du service. Correspond à la valeur de la propriété content d’une PDU.

[SSDU]

Session Service Data Unit - SDU de la couche session.

[context]

Information de contrôle de l’interface. Correspond à la valeur de la propriété context d’une IDU.

[PCI]

Protocol Control Information - Information de contrôle du protocole. Correspond à la combinaison des valeurs de la propriété header et de l’éventuelle propriété stamp d’une PDU.

[SDEK]

Symetrical data encryption key - Une clef de chiffrement de données symétrique.
