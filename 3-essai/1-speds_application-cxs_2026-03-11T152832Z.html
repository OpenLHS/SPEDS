6. Cas de Test

6.1. [CT_01] Requête de délégation bien formée

Hypothèses

S.O.

Données d’entrée
Identifiant Format Description

in_request

applicationInterface

Une requête de délégation bien formée de l’utilisateur de la couche

in_confirmation

AIDU

Une confirmation de délégation de la couche inférieure
Données de sortie
Identifiant Format Description

out_request

AIDU

Une requête de transfert à la couche inférieure

out_confirmation

applicationInterface

Une confirmation de délégation à l’utilisateur de la couche
Instructions

  1. Appel de la procédure [PRO-01] "Demande de délégation d’un message"

  2. Appel de la procédure [PRO-02] "Confirmation d’une délégation"

Conditions de réussite

  1. La requête de transfert à la couche inférieure out_request est bien formée :

    1. Pour l’attribut context :

      1. L’identifiant de projet PGA est celui indiqué dans projectId de in_request;

      2. Le code de la source source_code est celui indiqué dans sourceCode de in_request;

      3. Le code du destinataire destination_code est celui indiqué dans destinationCode de in_request;

      4. Le service service contient la valeur delegate;

      5. La primitive service_primitive contient la valeur request;

    2. Pour l’attribut header du PDU sérialisé dans message :

      1. Le type de message est celui indiqué dans msgType de in_request;

      2. L’identifiant message est celui indiqué dans msgId de in_request;

      3. Les informations de version contiennent celles fournies à l’initialisation de l’application;

    3. Le contenu de l’attribut content du PDU sérialisé dans message est identique à celui dans content de in_request

  2. La confirmation transmise out_confirmation est bien formée :

    1. Le service contient la valeur delegate;

    2. La primitive contient la valeur confirm;

    3. Le code de la source sourceCode est celui indiqué dans context → source_code de in_confirmation;

    4. Le code du destinataire destinationCode est celui indiqué dans context → destination_code de in_confirmation;

    5. L’identifiant de projet projectId est celui indiqué dans context → PGA de in_confirmation;

    6. L’identifiant de message msgId est celui indiqué dans le même champ de in_request;

    7. Le type de message msgType est celui indiqué dans le même champ de in_request;

    8. Le contenu content est identique à celui dans message de in_confirmation.

7. Essais

7.1. Environnement

Identifiant Description

ENV_01

  1. Les informations de version sont correctement transmis à l’entité lors de son initialisation;

  2. Le plan du PGA associé à l’identifiant de PGA contenu dans le contexte des messages est en cours.

ENV_02

  1. Les informations de version sont correctement transmis à l’entité lors de son initialisation;

  2. Le plan du PGA associé à l’identifiant de PGA contenu dans le contexte des messages n’est pas en cours.

7.2. Descriptions

Identifiant Environnement Cas de test Description

E_01

ENV_01

  • [CT_01] Requête de délégation bien formée

Une requête bien formée reçue de l’utilisateur de la couche et une confirmation de délégation reçue de la couche inférieure produisent une requête de transfert à la couche inférieure et une confirmation de délégation à l’utilisateur de la couche. Le contenu de la confirmation reçue est identique au contenu de la confirmation transmise.

E_02

ENV_01

  • [CT_02] Requête de délégation mal formée

Une requête mal formée reçue de l’utilisateur de la couche produit une confirmation échouée de délégation à l’utilisateur de la couche. Aucune requête de transfert à la couche inférieure n’est produite.

E_03

ENV_02

  • [CT_03] Requête de délégation pour un plan qui n’est pas en cours

Une requête bien formée reçue de l’utilisateur de la couche, mais qui contient un identifiant de projet associé à un PGA dont le plan n’est pas en cours, produit une confirmation échouée de délégation à l’utilisateur de la couche. Aucune requête de transfert à la couche inférieure n’est produite.

E_04

ENV_01

  • [CT_04] Confirmation d’une délégation qui ne correspond pas à une requête précédente

Une confirmation reçue de la couche inférieure qui ne correspond à aucune demande transmise et non confirmée ne produit aucun signal de confirmation.

E_05

ENV_01

  • [CT_05] Indication de message bien formée

Une indication de message bien formée reçue de la couche inférieure et une réponse de l’utilisateur de la couche produisent une indication de message à l’utilisateur de la couche et une réponse à la couche inférieure. Le contenu de la réponse reçue est identique au contenu de la réponse transmise.

E_06

ENV_01

  • [CT_06] Indication de message mal formée

Une indication mal formée reçue de la couche inférieure produit une réponse échouée à la couche inférieure. Aucune indication à l’utilisateur de la couche n’est produite.

E_07

ENV_02

  • [CT_07] Indication de message pour un plan qui n’est pas en cours

Une indication bien formée reçue de la couche inférieure, mais qui contient un identifiant de projet associé à un PGA dont le plan n’est pas en cours, produit une réponse échouée de délégation à la couche inférieure. Aucune indication à l’utilisateur de la couche n’est produite.

E_08

ENV_01

  • [CT_08] Réponse de réception de message qui ne correspond pas à une indication précédente

Une réponse reçue de l’utilisateur de la couche qui ne correspond à aucune indication transmise et non répondue ne produit aucun signal de réponse.
