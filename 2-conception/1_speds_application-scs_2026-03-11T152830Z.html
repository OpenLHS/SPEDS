1. Rôle du module

Le module « SPEDS - application » est d’offrir une couche de services de protocole pour échanger des messages applicatifs via des services et applications.

2. Modèle propre au module

Le module doit assurer :

  • Un mécanisme qui permet de traiter les signaux de confirmation (primitive de message confirm) et de réponse (primitive de message response) afin de déclencher les signaux associés.

Ceci peut être implémenté par n’importe quelle technologie y compris par un mécanisme de fonctions et de retour de fonctions.

  • Un mécanisme qui permet de vérifier le contenu d’un APDU à partir du type du message et avec l’aide de fonctions préinitialisées au démarrage du module.

Ceci peut être implémenté par un mécanisme de fonctions préinitialisées au module (voir annexe pour un exemple).

3. Type au module

3.1. Type simple de données

Table 4. Type simple de donnée
Nom Format Description

service

Une chaîne de caractères respectant l’énumération service défini dans AIDU et pouvant contenir les valeurs suivantes : DELEGATE et TRANSFER.

Nom d’un service.

primitive

Une chaîne de caractères respectant l’énumération service_primitive défini dans AIDU et pouvant contenir les valeurs suivantes : REQUEST, INDICATION, RESPONSE et CONFIRM.

Nom d’une primitive de service.

sourceCode

Une chaîne de caractères respectant le format de source_code défini dans AIDU.

Le code de l’entité agissant comme source

destinationCode

Une chaîne de caractères respectant le format de destination_code défini dans AIDU.

Le code de l’entité agissant comme destination.

projectId

Une chaîne de caractères respectant le format de PGA défini dans AIDU.

Identifiant de projet.

msgId

Une chaîne de caractères respectant le format de id défini dans APDU.

Numéro d’identification du message.

msgType

Une chaîne de caractères respectant l’énumération msgtype définie dans APDU et pouvant contenir les valeurs suivantes : PLAN et ADMINISTRATION.

Type du message applicatif.

content

Chaîne de caractères respectant le format JSON.

Contenu d’un message applicatif.

3.2. Type complexe de données

3.2.1. Application Interface

Interface entre l’utilisateur de la couche application et la couche application.

Un type représentant la sommation des types suivants : service, primitive, sourceCode, destinationCode, projectId, msgId, msgType et content.

3.2.2. Format d’entête (Ent_A)

Un type décrivant un format d’entête.

Ce type est attendu être équivalent à la propriété {header} appartenant au schéma JSON du PDU

(cf. annexe Schéma JSON du message de la couche application.).

Le PCI est l’équivalent de Ent_A.

3.2.3. Message du protocole de la couche application (APDU)

Un type décrivant un format de message. Ce type est attendu être équivalent au schéma JSON du même objet

(cf. annexe Schéma JSON du message de la couche application.).

3.2.4. Interface de l’unité de données du service de la couche application (AIDU)

Unité d’information transférée en une seule interaction à l’interface de 2 couches, constituée d’information de contrôle d’interface (ICI(N)) et d’unité de service de données (SDU(N)).

(cf. annexe Schéma JSON du format d’interface d’unité de données entre la couche application et présentation.).

3.3. Type de message

  • PLAN

  • ADMINISTRATION

4. Hypothèse du module

[HYP-001]

La version et la référence du protocole sont connues à l’initialisation du module.
[HYP-002]

Pour chaque type de message APDU, une liste de schémas de JSON y est associée.

5. Contrainte du module

[CON-001]

La couche application (incluant les états de celle-ci et les structures) doit gérer la concurrence.
[CON-002]

La couche application utilise speds-toolkit-library pour vérifier si un plan est en cours.

6.1. [PRO-01] Processus « Demande d’une délégation d’un message »

6.1.1. Interface

Déclencheur Condition

Interface reçue de l’utilisateur de la couche.

  • le service est delegate

  • la primitive est request
Flux d’entrée

  • une requête de délégation de l’utilisateur de la couche - applicationInterface

Flux de sorties

  • une requête de transfert à la couche inférieure - AIDU.

  • une confirmation de délégation à la couche supérieure - applicationInterface

6.1.2. Comportement

[PRO-01.1]

Vérification de l’interface

  • Si la vérification du contenu ne s’est pas fait avec succès alors le contenu de confirmation de l’interface est FAILED: syntax.

  • Si le plan n’est pas en cours alors le contenu de confirmation de l’interface est FAILED: ended plan.
[PRO-01.2]

Construction de l' AIDU si la vérification de l’interface a été fait avec succès

  • La ICI est construite avec

    • l’identifiant de projet est celui indiqué de applicationInterface reçue en entrée;

    • le code source est celui indiqué de applicationInterface reçue en entrée;

    • le code de destination est celui indiqué de applicationInterface reçue en entrée;

    • delegate comme service;

    • request comme primitive;

  • La APDU est construite avec

    • l’entête suivante:

      • le type de message est celui indiqué de applicationInterface reçue en entrée;

      • l’identifiant message est celui indiqué de applicationInterface reçue en entrée;

      • la version avec les informations définies à [HYP-001].

    • le contenu de applicationInterface reçue en entrée pour contenu.
[PRO-01.3]

Traitement avec la couche inférieure si un AIDU a été construit

  • Transmission de l' AIDU à la couche inférieure

  • Attente du signal du message de confirmation correspondant à cet AIDU.

  • Le contenu de confirmation de l’interface est le contenu du message de confirmation reçue.
[PRO-1.4]

Transmission de la confirmation de délégation à la couche supérieure.

  • La applicationInterface est construite avec

    • delegate comme service,

    • confirm comme primitive,

    • le code source est celui indiqué de l’interface reçue en entrée,

    • le code de destination est celui indiqué de l’interface reçue en entrée,

    • l’identifiant de projet est celui indiqué de l’interface reçue en entrée,

    • l’identifiant de message est celui indiqué de l’interface reçue en entrée,

    • le type de message est celui indiqué de l’interface reçue en entrée,

    • la contenu de confirmation de l’interface comme contenu.

6.1.3. Erreur

S.O.

6.2. [PRO-02] Processus « Confirmation d’une délégation »

6.2.1. Interface

Déclencheur Condition

Message reçu de la couche inférieure.

  • le service est delegate

  • la primitive est confirm
Flux d’entrée

  • une confirmation de délégation provenant la couche inférieure - AIDU

Flux de sorties

S.O.

6.2.2. Comportement

[PRO-02.1]

Signaler la confirmation

  • Si la confirmation correspond à une demande delegate.request transmise et non confirmée, signaler le message de confirmation;

  • sinon ignorer la confirmation.

6.2.3. Erreurs spécifiques

S.O.

6.2.4. Erreur

S.O.

6.3. [PRO-03] Processus « Indication de réception d’un message »

6.3.1. Interface

Déclencheur Condition

Message reçu de la couche inférieure

  • le service est transfer

  • la primitive est indication
Flux d’entrée

  • Une indication d’un message reçu par la couche inférieure - AIDU

Flux de sorties

  • une indication d’un message reçu à la couche supérieure - applicationInterface

  • une réponse vers la couche inférieure - AIDU

6.3.2. Comportement

[PRO-03.1]

Vérification de l' AIDU

  • Si la vérification du contenu ne s’est pas fait avec succès alors la réponse est FAILED: syntax.

  • Si le plan n’est pas en cours alors la réponse est FAILED: ended plan.
[PRO-3.2]

Transmettre l’interface à la couche supérieure si la vérification a été fait avec succès

  • Transmettre à la couche supérieure

    • L’interface applicationInterface avec

      • transfer comme service,

      • indication comme primitive,

      • le code source est celui indiqué de l' AIDU reçue en entrée,

      • le code de destination est celui indiqué de l' AIDU reçue en entrée,

      • l’identifiant de projet est celui indiqué de l' AIDU reçue en entrée,

      • l’identifiant de message est celui indiqué de l' AIDU reçue en entrée,

      • le type de message est celui indiqué de l' AIDU reçue en entrée,

      • le contenu de l’indication comme contenu.

  • Attente du signal de réponse correspondant à cette interface, et conserver la réponse.
[PRO-3.3]

Transmettre la réponse obtenue à la couche inférieure

  • transfer comme service;

  • response comme primitive;

  • un numéro de suivi dans les options qui est identique au numéro de suivi de l' AIDU reçue en entrée.

6.3.3. Erreur

S.O.

6.4. [PRO-04] Processus « Réponse de réception d’un message »

6.4.1. Interface

Déclencheur Condition

Interface reçue de l’utilisateur de la couche.

  • le service est transfer

  • la primitive est response
Flux d’entrée

  • une réponse de la couche supérieure (IDU) - applicationInterface

Flux de sorties

S.O

6.4.2. Comportement

[PRO-4.1]

Signaler la réponse

  • Si la réponse correspond à une demande transfer.indication transmise et non répondue, signaler la réponse;

  • sinon ignorer la réponse.

6.4.3. Erreur

S.O.

7. Incongruités

[INC-01] La spécification et le Schéma JSON du message de la couche application. définissent un type de message différent. La conception respecte le type de message défini par le Schéma JSON du message de la couche application..

10.1. Glossaire

Source

L’entité qui initie l’envoi de données.
Destination

L’entité qui reçoit les données.
Couche inférieure

La couche de communication qui offre des services utilisés par la couche application.

Unité de données du service.
ICI

Information de contrôle de l’interface.
PDU

Unité de données de protocole.

10.3. Exemple de fonction de vérification de contenu de APDU

  • Chaque type de message a un schéma JSON regroupe plusieurs schémas.

    • Celui-ci utilise oneOf pour que la désérialisation soit possible si un des schémas est respecté.

    • Celui-ci est initialisé au module.

  • La fonction vérifie que la désérialisation réussie à partir d’un type de message et un contenu de message.
