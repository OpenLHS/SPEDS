1. Rôle du module

Le rôle de SPEDS - Transport est d’assurer une communication de bout en bout. Entre autres, la couche transport a l’objectif de garantir l’intégrité des transmissions et asssurer la fiabilité des échanges.

2. Implémentation du module

Le module SPEDS - Transport est implementé par le produit speds-transport.

Pour l’itération 9, le module ne sera considéré que comme un simple système qui envoie des messages et attends une confirmation en ne vérifiant que l’intégrité des contenus. Il n’y aura donc aucun système de fenêtrage pour le moment. Les processus concernant ce mécanisme ne seront pas présents dans cette conception.

Pour les fins du concepteur qui aura à faire avancer cette conception en lien avec les concepts PFE/PFA, voir la version gardé dans le dossier d’archive _arc ou restaurer avec l’historique les sections en développement.

Ainsi, les types de message traité par la couche Transport sont TRA.MSG.ENV et TRA.MSG.REC. Tout autre message doit être ignoré.

Le module SPEDS - Transport consiste en l’implémentation de la couche Transport du protocole SPEDS qui s’inspire du modèle OSI de protocole par couche.

3. Modèle propre au module

3.1. État de transport

Le module requiert de valider par un mécanisme au choix qu’un message de confirmation reçu de la couche Réseau par une source ou un destinataire soit bien la confirmation d’une demande d’échange de données reçue précédemment de la couche Session.

Le module requiert de valider qu’une réponse reçue de la couche Session par une source ou un destinataire correspond à l’indication reçue précédemment de la couche Réseau, donc possède le même identifiant. Conséquemment, le module doit conserver un ensemble d’identifiants de messages envoyés jusqu’à l’obtention de la réponse correspondante. Il est possible que plus que l’identifiant soit conservé pour usage futur (ex. source_code et destination_code).

4. Mode de transmission

Pour les versions courantes, le mode de transmission est sans connexion avec accusé de réception. (Il sera orienté connexion lors de l’implémentation du mécanisme de fenêtre d’envoi.)

5.3. Type de message

Schéma JSON

Il faut suivre le schéma JSON en lien avec les messages de la couche Transport

(cf. annexe Protocol Data Unit de la couche transport (4)).
Il est à noter que seul les TRA.MSG sont traité pour le moment et les autres types sont ignorés.

5.3.1. TRA.MSG.ENV

Voici les particularités du message TRA.MSG.ENV de l'Unité de données de protocole de la couche Transport (TPDU):

  • Pour l’attribut header:

    • L’attribut msgtype contient la valeur TRA.MSG.ENV;

5.3.2. TRA.MSG.REC

Voici les particularités du message TRA.MSG.REC de l'Unité de données de protocole de la couche Transport (TPDU):

  • Pour l’attribut header:

    • L’attribut msgtype contient la valeur TRA.MSG.REC;

    • Le contenu est vide

6. Hypothèse du module

[HYP-001]

Chaque entité doit être identifiée par un code unique et joignable à une adresse réseau (IRI) unique, permettant ainsi la communication via la couche transport.
[HYP-002]

Les couches sous-jacentes à la couche Transport sont en mesure de transmettre les messages Transport de la source à la destination.

7. Contrainte du module

[CON-001]

La suite cryptographique (spécifiquement la fonction de hachage) est celle définie par la Politique de chiffrement par défaut pour le GRIIS.
[CON-002]

La couche transport (incluant les états de celle-ci et les structures) doit gérer la concurrence.

Création de l'Unité de données de protocole de la couche Transport (TPDU) de type TRA.MSG.ENV

  1. Création de l'Information de contrôle de protocole de la couche Transport (TPCI).

  3. Création du message Transport de type TRA.MSG.ENV contenant l'Information de contrôle de protocole de la couche Transport (TPCI), le sceau d’intégrité [SEAL] et le contenu qui se trouve être l'Unité de données de services (SDU) reçu en entrée.

[PRO-01.6]

La sortie est l'Unité de données d’interface de la couche Transport (TIDU) qui a été précédemment construit à [PRO-01.5].
[PRO-01.7]

Attendre la confirmation de la couche inférieure Réseau.

Réception d’une confirmation

[PRO-01.8]

Si la confirmation [PRO-01.7] est un échec, alors:

  1. Création de l'Unité de données d’interface de la couche Session (SIDU) composé de l'Information de contrôle d’interface de la couche Session (SICI) contenant les informations suivantes :

    1. Les adresses source_iri et destination_iri sont les mêmes que dans la ICI du message de confirmation

    2. Le code source_code est le code unique associé à source_iri de l'Information de contrôle d’interface de la couche Session (SICI) reçue en entrée,

    3. Le code destination_code est le code unique associé à destination_iri de l'Information de contrôle d’interface de la couche Session (SICI) reçue en entrée,

    4. La primitive de service service_primitive est 'confirm'.

  2. Le message est celui de la confirmation, i.e. FAILED: raison de l’échec.

  3. La sortie vers la couche Session est l'Unité de données d’interface de la couche Session (SIDU) qui a été précédemment construit et arrêt du processus.

Attente d’une réponse TRA.MSG.REC

[PRO-01.9]

Attendre le signalement de [PRO-02] Processus « Traite une réponse d’échange de données » que le TRA.MSG.REC a été reçu et préparer le message de confirmation pour la couche supérieure.
[PRO-01.10]

Création de l'Information de contrôle d’interface de la couche Session (SICI) contenant les informations suivantes :

  1. L’adresse source_iri est la destination_iri de l'Information de contrôle d’interface de la couche Transport (TICI) reçue en [PRO-02] Processus « Traite une réponse d’échange de données »,

  2. L’adresse destination_iri est la source_iri de l'Information de contrôle d’interface de la couche Transport (TICI) reçue en [PRO-02] Processus « Traite une réponse d’échange de données »,

  3. Le code source_code qu’on retrouve dans l'Unité de données de protocole de la couche Transport (TPDU),

  4. Le code destination_code qu’on retrouve dans l'Unité de données de protocole de la couche Transport (TPDU),

  5. La primitive de service service_primitive est confirm.
[PRO-01.11]

Création de l'Unité de données d’interface de la couche Session (SIDU) composé de l'Information de contrôle d’interface de la couche Session (SICI) précédemment créé [PRO-01.10] et de l'Unité de données de services (SDU) où le SDU est le résultat de la vérification [PRO-02.2], donc SUCCEED ou FAILED: information.
[PRO-01.12]

La sortie vers la couche Session est l'Unité de données d’interface de la couche Session (SIDU) qui a été précédemment construit à [PRO-01.11].
