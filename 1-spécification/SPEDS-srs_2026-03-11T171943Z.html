1. Généralités et présentations

1.1. Format d’une unité de données transférée entre chaque couche.

Chaque couche définit le contexte et le message transférés pour son interface avec la couche supérieur et inférieur. Il peut y avoir zéro ou plusieurs interfaces.

format d’une unité de données
{ "context" : object
, "message" : string
}

1.2. Conventions de notations

  1. La notation @K(X) indique le chiffrement avec la clé symétrique K du contenu X.

  2. La notation @@K(X) indique le déchiffrement avec la clé symétrique K du contenu X.

  3. La notation enc(X, PuK) indique le chiffrement asymétrique du contenu X avec la clé publique PuK.

  4. La notation enc(X, C), où C est un code d’une composante, indique le chiffrement asymétrique du contenu X avec la clé publique de C.

  5. La notation dec(X, PK) indique le déchiffrement asymétrique du contenu X avec la clé privée PK.

  6. La notation dec(X, C), où C est un code d’une composante, indique le déchiffrement asymétrique du contenu X avec la clé privée de C.

  7. La notation sig(X, PK) indique la signature (le chiffrement asymétrique) du contenu X avec la clé privée PK.

  8. La notation sig(X, C), où C est un code d’une composante, indique la signature (le chiffrement asymétrique) du contenu X avec la clé privée de C.

  9. La notation ver(X, PuK) indique la vérification (le chiffrement asymétrique) du contenu X avec la clé publique PuK.

  10. La notation ver(X, C), où C est un code d’une composante, indique la vérification (le chiffrement asymétrique) du contenu X avec la clé publique de C.

Une signature numérique utilisant un algorithme de chiffrement asymétrique (comme RSA) implique l’utilisation d’une empreinte (ou « haché ») du contenu signé.

1.3. Sécurité et sérialisation des messages

1.3.1. Primitives de cryptographie

Voir la Politique de chiffrement par défaut pour le GRIIS et l’hypothèse [HY-CY-01] des Hypothèses de la section Spécifications globales.

1.3.2. Sérialisation

  • Les objets sont sérialisés en JSON. Un schéma JSON défini alors les structures des objets JSON.

  • Les chaînes d’octets (bytestring) sont encodés dans les objets JSON en base64 en utilisant le jeu de caractère UTF-8.

1.4. Terminologie

1.4.1. Convention de notation pour les types de messages

Les types de messages de la couche Application (couche 1) sont des chaînes de caractères composées de deux parties : la première indique la catégorie de message et sont attachés au protocole utilisé ; la deuxième indique si le message est un envoi de message (REQUEST) ou de réponse (RESPONSE).

Dans les autres couches, la décomposition de la chaîne de caractères composant le type est sur le modèle LLL.CCC.III où LLL caractérise la couche (p.ex. TRA), CCC caractérise l’envoi du message (ENV) ou la réception d’une réponse (REC) et III donne une indication supplémentaire sur la catégorie du message. Ainsi,

  • INSTRUCTION.REQUEST est un message de la couche application qui envoie (REQUEST) une instruction à un CE (INSTRUCTION) ;

  • SES.MSG.ENV est un message de la couche session (SES) qui envoie (ENV) le message principal (MSG), c’est-à-dire le colis transmis par la couche supérieure ;

  • RES.MSG.ENV est un message de la couche réseau (RES) qui correspond à l’envoi (ENV) d’un message (MSG).

Certaines couches n’ont que la caractérisation ENV. Il en va ainsi de la couche Présentation et la couche Réseau.

1.4.2. Réponse

Entre deux entités S et D, pour une couche spécifique, un message est une réponse entre entités si le contenu répond à un envoi de l’autre entité. Puisque S initie l’envoi, la réponse est fourni par D.

Au sein d’une même entité, entre deux couches, un message est une réponse intercouche si, pour le protocole, il est défini comme la réponse à une transmission entre ses couches. Ce n’est pas le contenu qui définit la qualité de réponse.

Le terme réponse peut être utilisé seul si le contexte du texte permet de discerner entre les deux types de réponses.

1.4.3. Unité de données

Les échanges entre les entités se font à l’aide des unités de données de service (SDU). Au niveau de chaque couche de protocole, la SDU de la couche supérieure est encapsulée avec de l’information de contrôle du protocole (PCI) pour fournir l’unité de données du protocole (PDU) qui sera envoyées et traitées par la couche du même niveau de l’entité distante. Pour transmettre les PCU entre les couches, ces dernières sont accompagnée de leur information de contrôle de l’interface (ICI). Entre les couches, ce sont donc des unités de données d’interface (IDU) qui sont échangées.

[IDU]

Unité de données d’interface constituée d’une ICI et d’un PDU. La notation \((n)\)IDU dénote la IDU de la couche de niveau \((n)\). Une IDU contient toujours le service demandé et la primitive de service appelée sur ce service.
[ICI]

Information de contrôle de l’interface. Correspond à la valeur de la propriété context d’une IDU. La notation \((n)\)ICI dénote la ICI de la couche de niveau \((n)\).
[PDU]

Unité de données du protocole. Correspond à la valeur de la propriété message d’une IDU. La notation \((n)\)PDU dénote la PDU de la couche de niveau \((n)\).
[PCI]

Information de contrôle du protocole. Correspond à la combinaison des valeurs de la propriété header et de l’éventuelle propriété stamp d’une PDU. La notation \((n)\)PCI dénote la PCI de la couche de niveau \((n)\).
[SDU]

Unité de données du service. Correspond à la valeur de la propriété content d’une PDU. La notation \((n)\)SDU dénote la SDU de la couche de niveau \((n)\).

Les couches peuvent être identifiées par une lettre ou un chiffre  :

  1. A pour Application

  2. P pour Présentation

  3. S pour Session

  4. T pour Transport

  5. N pour Network

  6. L pour (Data)Link

En général, on retrouve les relations suivantes :

  • Pour une transmission d’une \((n)\)SDU par une \((n)\)IDU vers une couche \(n+1\) :

    • \((n)\)PDU \(=\) \((n)\)PCI + \((n)\)SDU

    • \((n)\)IDU \(=\) \((n)\)ICI + \((n)\)PDU

    • Dans ce cas, si la couche \(n+1\) veut retransmettre le colis reçu
      \((n+1)\)SDU \(=\) \((n)\)PDU

  • Pour une transmission d’une \((n)\)SDU par une \((n)\)IDU vers une couche \(n-1\) :

    • \((n)\)PDU \(=\) \((n)\)SDU

    • \((n-1)\)IDU \(=\) \((n-1)\)ICI + \((n)\)PDU

    • La fonction de hachage est SHA-512.

  • Couche Réseau (5)

    • la fonction de hachage utilisé est SHA-512 ;

    • l’algorithme de signature utilisé est RSA avec une taille des clés de 4096 bits ;

    • le format des certificats utilisé est X.509 .

3. Spécifications par couche

3.1. Spécification de la couche Application

3.1.1. Responsabilité

  1. La couche Application doit initier l’envoi entre source et destinataire d’un message initié par la source ainsi que de la réponse générée par le destinataire. Les messages sont envoyés avec pour le service delegate et la primitive request et sont reçus avec le service transfer et la primitive indication dans tous les cas de figure.

  2. La couche Application doit filtrer les messages qui n’ont pas la syntaxe attendue pour les services delegate et transfer. Si un message contient un colis qui n’est pas au format JSON attendu en fonction de son type, le message est ignoré. Dans ce cas : si la primitive est request, un message de primitive confirm avec le contenu "failed: syntax" est transmis en retour ; si la primitive est un indication, un message de primitive response avec le contenu "failed: syntax" est transmis en retour. Cette vérification s’applique aussi au contenu du message qui doit être vérifié en fonction du type (msgtype).

  3. La couche Application doit vérifier qu’un message ayant un type PLAN est associé à un projet dont le plan est en cours. Autrement, si le plan n’est pas en cours, le message est ignoré (n’est pas transmis). Dans ce cas : si la primitive est request, un message de primitive confirm avec le contenu "failed: ended plan" est transmis en retour ; si la primitive est un indication, un message de primitive response avec le contenu "failed: ended plan" est transmis en retour.

Hypothèses de travail

  1. Le contenu reçu par la couche inférieure est dans le cadre d’un projet actif.

  2. Le contenu reçu par la couche inférieure est d’un hôte authentifié pour le projet actif.

  3. Le contenu reçu par la couche inférieure est intègre.

Contraintes de maintenabilité

  1. Le système doit permettre d’ajouter ou de supprimer un schema de contenu (JSON) parmi les filtres autorisés avec le plus de facilité possible. C’est-à-dire en limitant les redémarrages et les activités de développement. Ainsi la fonction de vérification doit permettre de relier les types de messages définis (comme PLAN) à des schémas JSON dont au moins un doit permettre de valider le message. Par exemple, le système peut utiliser une table de lien dans un fichier TSV, des répertoires portant le nom des msgtype possibles ou une base de données tant que cela évite des redémarrages et des activités de développement au possible.

3.1.2. Interfaces

Couche supérieure

Correspond aux appels de l’hôte, c’est-à-dire l’entité appelante.

entrant

Un contenu définissant un message à transmettre par un type d’un message vers une entité externe, un numéro d’identification du message, le code du destinataire et une chaîne de caractère décrivant un objet JSON. Le format du contenu est alors défini dans les diagrammes de protocole de 101 à 109 (selon le type). Le contenu est accompagné de l’identifiant du PGA en cours, de l’identifiant de la source et de l’identifiant du destinataire.

Deux types de contenu peuvent être envoyés :

  • Un message à déléguer : déclenchant un delegate.request à la couche inférieure après les vérifications attendues ;

  • Une réponse à un message reçu indiquant sa réception : déclenchant un transfer.response vers la couche inférieure puis donnant à l’hôte l’accès au message reçu si celui-ci passe les vérifications attendues.

sortant

Un contenu sous la forme d’un objet JSON transmis à l’hôte. Ce contenu définit un message pour l’entité qui doit être traité. Ce contenu est accompagné de l’identifiant du PGA en cours, de l’identifiant de la source et de l’identifiant du destinataire.

Deux types de contenu peuvent être envoyés :

  • Un message à traiter : résultat d’un transfer.indication de la couche inférieure dans le cas où le message reçu passe les vérifications attendues ;

  • Une confirmation de délégation qui est soit le résultat reçu par la couche inférieure, soit un échec de validation de syntaxe (message "FAILED: syntax") ou de plan terminé (message "FAILED: ended plan").

Couche inférieure
entrant

Un message en provenance de la couche Présentation avec un contexte indiquant le code de source, le code du destinataire, le service et la primitive de service utilisés et l’identifiant du PGA. Le message doit respecter le format standard (décrit après) sinon le message est ignoré.

Services et primitives de service utilisés

  • transfer.indication : primitive utilisée par la couche inférieure pour indiquer qu’un message est reçu et doit être transféré. Contient le message.

  • delegate.confirm : primitive utilisée par la couche inférieure pour indiquer que le message de même numéro d’identification transmis par delegate.request est traité. Déclenche une confirmation à l’hôte qui contient la même information.

sortant

Un message pour la couche Présentation accompagné d’un contexte indiquant le code de la source, le code du destinataire, le service et la primitive de service utilisés et l’identifiant du PGA. Le message doit respecter le format standard (décrit après).

Services et primitives de service utilisés

  • delegate.request : primitive utilisée par la couche courante pour envoyer un message (le contenu) à la couche inférieure et demander qu’il soit traité et que son envoi soit délégué à la couche Session. Est déclenché par la demande de l’hôte.

  • transfer.response : primitive utilisée par la couche courante pour indiquer à la couche inférieure que le message obtenu par transfer.indication est traité (transmis à l’hôte). Peut être déclenché par une réponse de l’hôte (transfer.response), auquel cas le contenu est le même que celui transmis par l’hôte, ou, peut être déclenché suite à un problème de structure (message "FAILED: syntax") ou de plan terminé (message "FAILED: ended plan").

Format du contexte d’une communication avec la couche inférieure
{ "source_code": string
, "destination_code": string
, "PGA": string
, "service": string
, "service_primitive": string
, "options": object (actuellement : false)
}

3.1.3. Format des messages pour la couche Application

Un message est une chaîne de caractère représentant un objet JSON. Le format de l’objet JSON est défini comme suit.

format d’un message
{ "header" : object
, "content" : object
}
Format d’un entête
{ "msgtype" : string
, "id" : string
, "version" : { "number" : "x.x.x"
              , "reference" : "IRI"
              }
, "parameters" : object (actuellement : false)
}

La version est celle de la couche Application. La référence est un IRI documentant la version.

Types de message

La chaîne de caractères représentant le type d’un message vers une entité externe est l’une des suivantes :

Valeurs possibles de la propriété msgtype

  • PLAN

  • ADMINISTRATIVE

Numéro d’identification du message (id)

Le numéro d’identification du message est un numéro généré unique (uuid).

3.2. Spécification de la couche Présentation

3.2.1. Responsabilité

  1. La couche Présentation doit gérer le chiffrement et le déchiffrement des messages avec un algorithme de chiffrement symétrique.

  2. La couche Présentation doit créer une clé de chiffrement symétrique (SDEK) si elle reçoit un message par la primitive request.

3.2.2. Interfaces

Couche supérieure
entrant

Un contenu (chaîne de caractère) en provenance d’une couche Application et un contexte indiquant le code de la source, le code du destinataire, le service, la primitive de service et l’identifiant du PGA.

Services et primitives de service utilisés

  • delegate.request : primitive utilisée par la couche supérieure pour envoyer un message à la couche Présentation et demander qu’il soit traité (chiffrement) et que son envoi soit délégué à la couche Session. Attends un delegate.confirm en réponse.

  • transfer.response : primitive utilisée par la couche supérieure pour indiquer à la couche Présentation que le message obtenu par transfer.indication est traité. Déclenche un transfer.response à la couche inférieure.

sortant

Un message de la couche Application (chaîne de caractère décrivant un objet JSON) et son contexte indiquant le code de la source, le code du destinataire, le service, la primitive de service et l’identifiant du PGA.

Services et primitives de service utilisés

  • delegate.confirm : primitive utilisée par la couche courante pour indiquer que le message transmis par delegate.request est traité. Est déclenché par un delegate.confirm de la couche inférieure ou par une erreur local. Dans ce dernier cas, le contenu doit alors être de la forme "FAILED: information").

  • transfer.indication : primitive utilisée par la couche Présentation pour indiquer qu’un message est reçu et doit être transféré. Contient le message. Est déclenché par un transfert.indication de la couche inférieure.

format du contexte d’une communication avec la couche supérieure
{ "source_code": string
, "destination_code": string
, "PGA": string
, "service": string
, "service_primitive": string
, "options": object (actuellement : false)
}
Couche inférieure
entrant

Un message en provenance de la couche Session avec un contexte indiquant le code de la source, le code du destinataire, l’identifiant du PGA, le service, la primitive de service et la clé de chiffrement utilisée (SDEK). Le message doit respecter le format standard (décrit après) sinon le message est ignoré.

Services et primitives de service utilisés

  • delegate.confirm : primitive utilisée par la couche inférieure pour indiquer que le message transmis par delegate.request est traité. Déclenche un delegate.confirm à la couche supérieure.

  • transfer.indication : primitive utilisée par la couche inférieure pour indiquer qu’un message est reçu et doit être transféré. Contient le message.

sortant

Un message pour la couche Session accompagné d’un contexte indiquant le code de la source, le code du destinataire, l’identifiant du PGA, le service, la primitive de service et la clé de chiffrement utilisé (SDEK). Le message doit respecter le format standard (décrit après).

Services et primitives de service utilisés

  • delegate.request : primitive utilisée par la couche Présentation pour envoyer un message à la couche inférieure pour que son envoi soit délégué à la couche Session. Est déclenché par un delegate.request de la couche supérieure.

  • transfer.response : primitive utilisée par la couche Présentation pour indiquer à la couche inférieure que le message obtenu par transfer.indication est traité. Peut être déclenché par une réponse de la couche supérieure (transfer.response), auquel cas le contenu est le même que celui transmis par la couche supérieure, ou par une erreur locale.   • delegate.request : primitive utilisée par la couche supérieure pour envoyer un message à la couche Session et demander qu’il soit envoyé à son destinataire. Attends un delegate.confirm en réponse.

  • transfer.response : primitive utilisée par la couche supérieure pour indiquer à la couche Session que le message obtenu par transfer.indication est traité. Déclenche un transfer.response à la couche inférieure.

sortant

Un message pour la couche Présentation accompagné d’un contexte indiquant le code de la source, le code du destinataire, l’identifiant du PGA, la primitive de service et la clé de chiffrement utilisé (SDEK). Le message doit respecter le format standard (décrit après).

Services et primitives de service utilisés

  • delegate.confirm : primitive utilisée par la couche courante pour indiquer que le message transmis par delegate.request est pris en charge (avant même son envoi). Est déclenché par un delegate.confirm de la couche inférieure ou par une erreur local. Dans ce dernier cas, le contenu doit alors être de la forme "FAILED: information").

  • transfer.indication : primitive utilisée par la couche Session pour indiquer qu’un message est reçu et doit être transféré. Contient le message. Est déclenché par un transfert.indication de la couche inférieure.

3.4. Spécification de la couche Transport

3.4.1. Responsabilité

  1. La couche Transport doit gérer le transport de bout en bout entre une source (identifiée par son code et sa IRI) et un destinataire. C’est-à-dire que la couche assure

    1. la garantie de l’intégrité des transmissions ;

    2. l’absence de perte de message (fiabilité), qui comprend

      1. la confirmation de la réception par l’entité ciblée,

      2. la répétition des envois au besoin,

      3. la gestion des répétitions (un message n’est transmis pour traitement qu’une fois à la couche supérieure) ;

Le système de fenêtrage n’est pas implémenté dans la version v7.0.0 de SPEDS. Ceci implique que les trois derniers points (1.b.ii et 1.b.iii) ne sont pas gérés actuellement.

3.4.2. Interfaces

Couche supérieure
entrant

Un message en provenance de la couche Session avec un contexte indiquant, le service, la primitive de service, le code de la source, le code du destinataire, l’adresse (IRI) de la source et l’adresse (IRI) du destinataire.

  • transfer.request : primitive utilisée par la couche supérieure pour envoyer un message à la couche Transport et demander qu’il soit traité et transféré. Attends un transfer.confirm en réponse.

  • transfer.response : primitive utilisée par la couche supérieure pour indiquer à la couche Transport que le message obtenu par transfer.indication est traité. Déclenche un message de réponse à la source.

sortant

Un message pour la couche Session (chaîne de caractère décrivant un objet JSON) avec son contexte indiquant, le service, la primitive de service, le code de la source, le code du destinataire, l’adresse (IRI) de la source et l’adresse (IRI) du destinataire.

Services et primitives de service utilisés

  • transfer.confirm : primitive utilisée par la couche Transport pour indiquer que le message transmis par transfer.request est traité. Est déclenché lorsque la confirmation est reçu du destinataire ou par une erreur local. Dans ce dernier cas, le contenu doit alors être de la forme "FAILED: information").

  • transfer.indication : primitive utilisée par la couche Transport pour indiquer qu’un message est reçu et doit être transféré. Contient le message. Est déclenché par un transfer.indication de la couche inférieure.

format du contexte d’une communication avec la couche supérieure
{ "source_code": string
, "destination_code": string
, "source_iri": string
, "destination_iri": string
, "service_primitive": string
, "options": object (actuellement : false)
}
Couche inférieure
entrant

Un message en provenance de la couche Réseau avec son contexte indiquant, le service, la primitive de service, l’adresse (IRI) de la source et l’adresse (IRI) du destinataire. Le message doit respecter le format standard (décrit après) sinon le message est ignoré.

Services et primitives de service utilisés

  • transfer.confirm : primitive utilisée par la couche inférieure pour indiquer que le message transmis par transfer.request est traité. Si le traitement indique un échec (FAILED: xxx), la demande de transfert de la couche supérieure qui a déclenché la demande de transfert de la couche

  • transfer.indication : primitive utilisée par la couche inférieure pour indiquer qu’un message est reçu et doit être transféré. Contient le message.

sortant

Un message pour la couche Réseau avec son contexte indiquant, le service, la primitive de service, l’adresse (IRI) de la source et l’adresse (IRI) du destinataire.

Services et primitives de service utilisés

  • transfer.request : primitive utilisée par la couche Transport pour envoyer un message à la couche inférieure pour que son envoi soit délégué à la couche Session. Est déclenché par un transfer.request de la couche supérieure ou pour transmettre une confirmation.

  • transfer.response : primitive utilisée par la couche Transport pour indiquer à la couche inférieure que le message obtenu par transfer.indication est traité. Peut être déclenché après les validations d’intégrités, auquel cas le contenu est "SUCCEED", ou par une erreur locale. Dans ce dernier cas, le contenu doit alors être de la forme "FAILED: information"). Par exemple, un échec de la validation de l’intégrité de l’entête pourrait déclencher un contenu "FAILED: integrity of the header is compromised".

format du contexte d’une communication avec la couche inférieure
{ "source_iri": string
, "destination_iri": string
, "service_primitive": string
, "options": object (actuellement : false)
}

3.4.6. Système de fenêtrage

États du module
Les inégalités sont à considérer dans l’anneau \(\mathbb{Z}/n\mathbb{Z}\) où \(n = ppcm(TFE, TFA)\).
État en tant que source

Chaque entité utilisant une couche transport doit conserver pour chaque destinataire potentiel (déterminé par son IRI), les informations suivantes :

  • PFE+ : les paramètres de la fenêtre d’envoi augmentés

    • PFE : les paramètres de la fenêtre d’envoi

      • DMR : le dernier numéro de message d’une série de messages répondus

      • DME : le dernier numéro de message d’une série de messages envoyés

      • TFE : la taille de la fenêtre d’envoi, c’est-à-dire le nombre de messages pouvant être envoyé avant de recevoir leurs réponses

    • Mémorisation des numéros de messages répondus supérieurs ou égales à DMR+1 (et inférieurs ou égales à DME)

  • Période de validité d’une entente \(\Delta\)

  • Période de validité d’une négociation \(\delta\)

  • Activation d’une négociation

État en tant que destinataire

L’entité doit aussi conserver pour chaque source potentielle (déterminé par son IRI), les informations suivantes:

  • PFA+ : les paramètres de la fenêtre d’acceptation augmentés

    • PFA : les paramètres de la fenêtre d’acceptation

      • DRC : le dernier numéro de message d’une série de messages dont la réponse a été confirmée

      • DMT : le dernier numéro de message d’une série de messages traités

      • TFA : la taille de la fenêtre d’acceptation, c’est-à-dire le nombre de messages pouvant être traité en attente de voir leur réponse confirmée par la source

    • Mémorisation des numéros de messages traités supérieurs ou égales à DMT+1 (et inférieurs ou égales à DRC+TFA)

  • Période de validité d’une entente \(\Delta\)

  • Période de validité d’une négociation \(\delta\)

  • Activation d’une négociation

Considération sur la concurrence

Il est important que toutes procédures qui débutent mettent en place un verrou sur les états (en tant que source ou destinataire) afin d’assurer que aucune situation de compétition (race condition) ne surviennent. L’état est considéré comme une section critique.

Détermination des durées

Supposons que \(\delta_d\) soit la durée impartie avant d’envoyer une répétition à un message non répondu. Soit R# le nombre de répétitions avant de considérer que l’envoi d’un message a échoué définitivement.

La durée de la validité d’une entente \(\Delta_d\) devrait être supérieur à \(\delta_d R\#\).

Le nombre de répétitions consécutives d’une négociation r# devrait être inférieur strictement à R#.

Pour SPEDS 6.0 :

  • \(\delta_d = 3s\)

  • \(R\# = 10\)

  • \(r\# = 5\)

  • \(\Delta_d = \delta_d R\# = 30s\)
Invariant

  • DMR ≤ DME ≤ DMR + TFE

  • Au moment d’un envoi (y compris une répétition), ME# vérifie que ME# ≤ DME.

  • DRC ≤ DMT ≤ DRC + TFA

  • DMT ≥ DMR

  • DMR ≥ DRC

Les opérations sur les états

  • Soit \({x}\) un entier, \(\overline{x}\) est le représentant de \({x}\) dans l’anneau \(\mathbb{Z}/n\mathbb{Z}\) où \(n = ppcm(TFE, TFA)\).
    C’est-à-dire que \(0 \leq \overline{x} \leq n\) et \(\exists k. k\in\mathbb{Z} \land x = \overline{x} + k n\).

  • Un numéro m est dans la fenêtre d’envoi si et seulement si
    \(0 < \overline{(m-DMR)} \leq TFE\)

  • Un numéro m est dans la fenêtre d’acceptation si et seulement si
    \(0 < \overline{(m-DRC)} \leq TFA\)

  • Mise à jour des PFE+ avec les PFA

    • Si le DMT > DMR alors DMR = DMT

    • Réajuster les numéros de messages répondus mémorisés

  • Mise à jour des PFA+ avec les PFE

    • Si le DMR > DRC alors DRC = DMR

    • Réajuster les numéros de messages traités mémorisés

  • Ajustement des PFE+ après un message répondu

    • Réajuster les numéros de messages répondus mémorisés

  • Ajustement des PFA+ après un message traité

    • Réajuster les numéros de messages traités mémorisés

  • Être dans la période de validité d’une entente

  • Être dans la période de validité d’une négociation

Comportement
De façon périodique, l’entité doit vérifier pour chaque source et chaque destinataire avec qui il communique si la période de validité d’entente est dépassée, c’est-à-dire si le temps écoulé depuis \(\Delta_m\) dépasse deux fois \(\Delta_d\). Dans ce cas, les paramètres correspondants (les PFE+ ou les PFA+) peuvent être réinitialisés.

3.5. Spécification de la couche Réseau

3.5.1. Responsabilité

  1. La couche Réseau doit garantir l’envoi entre chaque nœud pour parcourir un chemin entre la source (identifié par sa IRI) et le destinataire (identifié par sa IRI).

  2. La couche Réseau doit déterminer lors d’un transfer vers la couche inférieur quel est le prochain nœud à joindre (propriété destination_iri du NIDU).

3.5.2. Interfaces

Couche supérieure
entrant

Un message en provenance de la couche Transport avec un contexte indiquant, le service, la primitive de service utilisée, l’adresse (IRI) de la source initiale du contenu et l’adresse (IRI) du destinataire final du contenu.

Services et primitives de service utilisés

  • transfer.request : primitive utilisée par la couche supérieure pour envoyer un message à la couche Réseau et demander qu’il soit traité et transféré. Attends un transfer.confirm en réponse.

  • transfer.response : primitive utilisée par la couche supérieure pour indiquer à la couche Réseau que le message obtenu par transfer.indication est traité. Déclenche un transfer.response à la couche inférieure.

sortant

Un message pour la couche Transport (chaîne de caractère décrivant un objet JSON) avec un contexte indiquant, le service, la primitive de service utilisée, l’adresse (IRI) de la source initiale du contenu et l’adresse (IRI) du destinataire final du contenu.

Services et primitives de service utilisés

  • transfer.confirm : primitive utilisée par la couche Réseau pour indiquer que le message transmis par transfer.request est traité. Est déclenché par un transfer.confirm de la couche inférieure ou par une erreur local. Dans ce dernier cas, le contenu doit alors être de la forme "FAILED: information").

  • transfer.indication : primitive utilisée par la couche Réseau pour indiquer qu’un message est reçu et doit être transféré. Contient le message. Est déclenché par un transfer.indication de la couche inférieure.

format du contexte d’une communication avec la couche supérieure
{ "source_iri": string
, "destination_iri": string
, "service": string
, "service_primitive": string
, "options": object (actuellement : false)
}
Couche inférieure
entrant

Un message en provenance de la couche Liaison avec un contexte indiquant l’adresse (IRI) du nœud où il doit être transmis, le service et la primitive de service utilisée. Le message doit respecter le format standard (décrit après) sinon le message est ignoré.

Services et primitives de service utilisés

  • transfer.confirm : primitive utilisée par la couche inférieure pour indiquer que le message transmis par transfer.request est traité. Déclenche un transfer.confirm à la couche supérieure.

  • transfer.indication : primitive utilisée par la couche inférieure pour indiquer qu’un message est reçu et doit être transféré. Contient le message.

sortant

Un message pour la couche Liaison avec un contexte indiquant l’adresse (IRI) du nœud où le message doit être transmis, le service et la primitive de service utilisée.

Services et primitives de service utilisés

  • transfer.request : primitive utilisée par la couche Réseau pour envoyer un message à la couche inférieure pour qu’il soit transféré à son destinataire. Est déclenché par un transfer.request de la couche supérieure

  • transfer.response : primitive utilisée par la couche Réseau pour indiquer à la couche inférieure que le message obtenu par transfer.indication est traité. Peut être déclenché par une réponse de la couche supérieure (transfer.response), auquel cas le contenu est le même que celui transmis par la couche supérieure, ou par une erreur locale. Dans ce dernier cas, le contenu doit alors être de la forme "FAILED: information"). Par exemple, un échec d’authentification du nœud précédent pourrait déclencher un contenu "FAILED: cannot authentificate previous node".

format du contexte d’une communication avec la couche inférieure
{ "destination_iri" : string
, "service": string
, "service_primitive": string
, "options": object (actuellement : false)
}

3.6. Spécification de la couche Liaison

3.6.1. Responsabilités

  1. La couche Liaison doit transmettre le paquet reçu à un destinataire (désigné par destination_iri) en suivant un protocole établi.

  2. Pour v7.x.x, la couche Liaison ne devrait utiliser que le protocole HTTPS.

3.6.2. Interfaces

Couche supérieure
[entrant]

un message en provenance de la couche Réseau avec un contexte indiquant, la primitive de service utilisée, l’adresse (IRI) du nœud N destinataire et des éventuelles options.

Services et primitives de service utilisés

  • transfer.request : primitive utilisée par la couche supérieure pour envoyer un message à la couche Liaison et demander qu’il soit transmis au nœud suivant selon le protocole attendu. Attends un transfer.confirm en réponse.

  • transfer.response : primitive utilisée par la couche supérieure pour indiquer à la couche Liaison que le message obtenu par transfer.indication est traité. Est ignoré.

    [sortant]

    un message pour la couche Réseau (chaîne de caractère décrivant un objet JSON) avec un contexte indiquant, la primitive de service utilisée, l’adresse (IRI) du destinataire (c’est-à-dire le nœud actuel) et des éventuelles options.

Services et primitives de service utilisés

  • transfer.confirm : primitive utilisée par la couche Liaison pour indiquer que le message transmis par transfer.request est transmis au nœud suivant. Si une erreur se déroule, le contenu est "FAILED" éventuellement suivi d’une chaîne contenant de l’information (p.ex. "FAILED: unable to join the next node").

  • transfer.indication : primitive offerte par la couche Liaison pour indiquer qu’un message est reçu et doit être transféré. Contient le message.
