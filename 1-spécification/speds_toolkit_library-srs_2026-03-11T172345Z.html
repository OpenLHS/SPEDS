1. Définitions

  1. [DE-GE-01] Le module définit la liste de catégorie d’algorithme :

    1. algorithme de hachage ;

    2. algorithme de chiffrement symétrique ;

    3. algorithme de chiffrement asymétrique ;

    4. algorithme d’échange de clé de Diffie-Hellman ;

    5. algorithme de signature.

2. Contraintes

2.1. Conception (catégorie CO)

  1. [CO-CO-001] Le module est développé en utilisant le langage de programmation Java.

2.2. Performance (catégorie US)

2.3. Convivialité (catégorie US)

  1. [CO-US-001] La liste des choix des primitives cryptographiques pour chaque catégorie et chaque couche doit être paramétrable par le biais d’un fichier de configuration. Les catégories sont celles définies dans [DE-GE-01].

3. Hypothèses

  1. [HY-GE-01] L’entité utilisant SPEDS dispose des fonctions de cryptographie, de sécurité et générales qui peuvent être utilisées pour les exigences [EX-CY-xx], [EX-SE-xx] et [EX-GE-xx].

  2. [HY-GE-02] Une liste des fonctions de cryptographies acceptables pour le gestionnaire de l’entité utilisant SPEDS est disponible et accessible au module.

  3. [HY-GE-03] Les fonctions exigées par SPEDS sont disponibles dans cette liste :

    1. l’algorithme de hachage SHA-512 ;

    2. l’algorithme de chiffrement symétrique AES256-GCM ;

    3. l’algorithme de chiffrement asymétrique RSA-4096 ;

    4. l’algorithme d’échange de clé de Diffie-Hellman X25519.

4. Exigences

4.1. Les fonctions de service général

  1. [EX-GE-001] Dans le cadre de l’exécution d’un projet, le système doit fournir une fonction permettant de récupérer le IRI d’une entité inscrite dans le projet.

  2. [EX-GE-002] Dans le cadre de l’exécution d’un projet, le système doit fournir une fonction permettant de vérifier la légitimité d’une entité définie par un couple (code, puK) où code est le code identifiant l’entité (p. ex. GUID de Lodad) et puK est une clé de chiffrement asymétrique (considérée comme la clé publique de l’entité à légitimer).

  3. [EX-GE-003] Dans le cadre de l’exécution d’un projet, le système doit fournir une fonction permettant de récupérer la clé publique utilisée par une entité du projet définie par son code identifiant (p. ex. un GUID d’entité de Lodad).

  4. [EX-GE-004] Dans le cadre de l’exécution d’un projet, le système devrait considérer que, si puK est la clé publique obtenue pour l’entité code (p. ex. un GUID de Lodad) par la fonction définie dans l’exigence définie en [EX-GE-003] alors le couple (code, puK) est légitime par la fonction définie en [EX-GE-002].

  5. [EX-GE-005] Le système doit fournir une fonction permettant de vérifier si un projet est actif.

  6. [EX-GE-006] Dans le cadre de l’exécution d’un projet, le système doit fournir une fonction permettant de vérifier si le plan du projet est en cours.

4.2. Les fonctions de service de sécurité

  1. [EX-SE-001] Le système doit fournir une fonction permettant de valider un certificat pour une entité proprement identifiée par une IRI.

4.3. Les fonctions de service cryptographique

  1. [EX-CY-001] Le système doit offrir une fonction de chiffrement symétrique en fonction de la couche qui le demande.

  2. [EX-CY-002] Le système doit offrir une fonction de déchiffrement symétrique en fonction de la couche qui le demande.

  3. [EX-CY-003] Le système doit offrir une fonction de génération d’une clé pour le chiffrement symétrique en fonction de la couche qui le demande.

  4. [EX-CY-004] Le système doit offrir une fonction de chiffrement asymétrique en fonction de la couche qui le demande.

  5. [EX-CY-005] Le système doit offrir une fonction de déchiffrement asymétrique en fonction de la couche qui le demande.

  6. [EX-CY-006] Le système doit offrir une fonction de hachage en fonction de la couche qui le demande.

  7. [EX-CY-007] Le système doit offrir une fonction de signature en fonction de la couche qui le demande.

  8. [EX-CY-008] Le système doit offrir une fonction de vérification de signature en fonction de la couche qui le demande.

  9. [EX-CY-009] Le système doit offrir une fonction pour effectuer le choix d’une valeur dans un algorithme de poignée de main suivant la méthode Diffie-Hellman en fonction de la couche qui le demande.

  10. [EX-CY-010] Le système doit offrir une fonction pour déterminer la clé secrète dans un algorithme de poignée de main suivant la méthode Diffie-Hellman en fonction des choix effectué par les deux entités en communication et de la couche qui le demande.

  11. [EX-CY-011] Le système doit offrir à chaque couche une fonction qui indique le nom de l’algorithme disponible (sous une version standardisé) pour une catégorie définie en [DE-GE-01].

  12. [EX-CY-012] Le système doit offrir à chaque couche une fonction qui indique les paramètres par défaut de l’algorithme disponible pour une catégorie définie en [DE-GE-01].
